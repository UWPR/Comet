# Peptide Index (PI_DB) search performance review

Date: 2026-07-13
Scope: `CometSearch/CometPeptideIndex.cpp`, `CometSearch/CometSearch.cpp` (PI_DB code
paths), `CometSearch/search/PiStrategy.cpp`, `CometSearch/CometSearchManager.cpp`
(`InitializeSingleSpectrumSearch` / `DoSingleSpectrumSearchMultiResults`),
`CometWrapper/CometWrapper.cpp`. Focus is the real-time single-spectrum search
interface (`CometSearchManagerWrapper::DoSingleSpectrumSearchMultiResults` ->
`CometSearchManager::DoSingleSpectrumSearchMultiResults` -> `CometSearch::RunSearch(Query*)`
-> `CometSearch::SearchPeptideIndex(Query*, bool*)` -> `CometSearch::AnalyzePeptideIndex(Query*, ...)`),
with the batch PI_DB path reviewed for comparison since it shares the same scoring
core.

No code was changed for this review; this is a read-only survey of optimization
opportunities, ranked by expected impact.

## 1. `AnalyzePeptideIndex` takes `DBIndex` by value in the hottest loop (high impact)

**Status: Done.** Both overloads now take `const DBIndex&`. No caller changes were
needed (both call sites already passed lvalues). Verified with a full build and the
19-test unit suite (T19/T20 passing, exercising this exact path).

`CometSearch.h:280-289` declares both overloads as:

```cpp
void AnalyzePeptideIndex(int iWhichQuery, DBIndex sDBI, bool* pbDuplFragment, struct sDBEntry* dbe);
static void AnalyzePeptideIndex(Query* pQuery, DBIndex sDBI, bool* pbDuplFragment, struct sDBEntry* dbe);
```

`DBIndex` (`CometSearch/core/Types.h:219-227`) contains `vector<char> pcVarModSites`
alongside a `char sPeptide[MAX_PEPTIDE_LEN]` (51 bytes). Passing it by value means
every call copy-constructs the struct, including the `vector<char>`. For any
modified peptide (`pcVarModSites` non-empty), that copy performs a heap allocation
and a `memcpy`, then a `free` when the by-value parameter goes out of scope --
inside the innermost per-candidate loop.

The RTS caller, `SearchPeptideIndex(Query*, bool*)` (`CometSearch.cpp:2014-2059`),
calls this for every `g_pvDBIndex` entry inside the precursor mass-tolerance window:

```cpp
for (size_t i = iStart; i < g_pvDBIndex.size(); ++i)
{
   ...
   AnalyzePeptideIndex(pQuery, g_pvDBIndex[i], pbDuplFragment, &dbe);   // CometSearch.cpp:2049
}
```

For a wide precursor tolerance or a dense mass region this is dozens to hundreds of
candidates per spectrum, each paying an avoidable malloc/free pair whenever the
peptide carries a variable mod. This is pure, unconditional overhead added to the
single-spectrum RTS latency path, which is exactly where allocation-free code
matters most.

**Fix**: change both `AnalyzePeptideIndex` overloads to take `const DBIndex& sDBI`.
Nothing in the function mutates `sDBI` (it only reads `sPeptide`, `pcVarModSites`,
`dPepMass`, `cPrevAA`/`cNextAA`, `lIndexProteinFilePosition`), so this is a
signature-only change with no behavioral effect.

## 2. Batch PI_DB search does not use the thread pool at all (high impact for batch mode)

**Status: Done**, together with #3 below (they required the same rewrite). A new
shared helper, `CometSearch::EnsurePeptideIndexLoaded()`, does the
once-only, double-checked-locking load of `g_pvDBIndex`/`g_pvProteinsList`
(via `ReadPeptideIndex()`), re-initializes fragment/parent masses from the `.idx`
header, and creates the AScorePro interface if `iPrintAScoreProScore` is set (this
last step previously only happened in the old batch path; RTS PI_DB + AScorePro
now gets it too as a side benefit). `RunSearch(Query*)`'s PI_DB branch and the
rewritten `SearchPeptideIndex(ThreadPool*, vector<Query*>&)` both call it.
`SearchPeptideIndex(ThreadPool*, vector<Query*>&)` now submits one `tp->doJob()`
per query -- each job acquires a memory-pool slot and calls the thread-local
`SearchPeptideIndex(Query*, bool*)` overload -- exactly mirroring the FI_DB batch
dispatch. Net: batch PI_DB search now scales with `iNumThreads` instead of running
on a single core. Validated with the full suite, including T20 (PI_DB batch +
AScorePro) and the T17/T18 integration tests (index build count and
build-to-build determinism unaffected).

`CometSearch::RunSearch(int, int, ThreadPool* tp, vector<Query*>&)`
(`CometSearch.cpp:250-255`) dispatches PI_DB batch searches to:

```cpp
CometSearch sqSearch;
sqSearch.SearchPeptideIndex(tp, queries);
```

but the target function's signature is `SearchPeptideIndex(ThreadPool* /*tp*/, vector<Query*>& queries)`
(`CometSearch.cpp:1808`) -- `tp` is explicitly discarded. The whole function is a
single serial pass over the `.idx` file: it opens one `FILE*`, seeks through the
on-disk mass index, and walks entries with `fread`-based `ReadPeptideIndexEntry`
calls (`CometSearch.cpp:1962-2003`), scoring one candidate at a time on the calling
thread only.

Compare with the FI_DB batch path immediately above it (`CometSearch.cpp:200-248`),
which submits one `doJob()` per query to the pool and scores queries concurrently.
PI_DB batch searches get none of that: on an N-core machine, a batch PI_DB search
(`comet -search` against a `.idx` peptide index, e.g. via `PiStrategy::executeBatch`)
uses exactly one core for the entire scoring phase, regardless of `iNumThreads`.

**Fix**: parallelize the batch path the same way FI_DB does -- submit one job per
query (or a chunk of queries) to `tp`, with each job calling the existing
thread-local `SearchPeptideIndex(Query*, bool*)` / `AnalyzePeptideIndex(Query*, ...)`
overloads against the already-loaded, read-only `g_pvDBIndex` in memory (see next
finding) instead of streaming from disk. This also removes the need for the
separate file-scanning implementation entirely.

## 3. Batch path duplicates a slower, disk-streaming search instead of reusing `g_pvDBIndex` (medium-high impact, also a maintainability issue)

**Status: Done**, as part of the #2 rewrite. The entire serial disk-streaming
`SearchPeptideIndex(ThreadPool*, vector<Query*>&)` body (file open, on-disk mass
index seek/read, per-entry `fread`-based `ReadPeptideIndexEntry` streaming,
`BinarySearchMass`-against-`queries` matching) was replaced; batch mode now shares
the exact same in-memory, mass-sorted `g_pvDBIndex` binary-search path the RTS
overload uses. The old file-based overload and the now-orphaned
`AnalyzePeptideIndex(int iWhichQuery, ...)` it called (~560 lines, see the bonus
finding above) have both been deleted. Net diff across `CometSearch.cpp`,
`CometSearch.h`, and `CometSearchManager.cpp` for #1-#5 combined: 118 insertions,
797 deletions.

The codebase already has two independent implementations of "find candidates within
a mass window and score them":

- `SearchPeptideIndex(Query* pQuery, bool* pbDuplFragment)` (`CometSearch.cpp:2014`):
  binary search over the fully in-memory, mass-sorted `g_pvDBIndex` vector, then a
  linear scan bounded by the tolerance window. No file I/O per query.
- `SearchPeptideIndex(ThreadPool*, vector<Query*>& queries)` (`CometSearch.cpp:1808`):
  re-opens the `.idx` file, re-parses the header (skipped only if
  `g_bPeptideIndexRead`), reads the on-disk mass-bucket index, and then issues one
  `fread`-based `ReadPeptideIndexEntry` per candidate peptide via `comet_fseek`/`fread`
  syscalls, for the entire query batch combined (not per query -- the loop advances
  through the file once, matching each entry against whichever queries currently
  overlap its mass via `BinarySearchMass`).

The second path exists only for batch mode and never touches `g_pvDBIndex`, even
though `ReadPeptideIndex()` (`CometPeptideIndex.cpp:43`) already loads the entire
index into that in-memory, sorted vector during `InitializeSingleSpectrumSearch()`
for the RTS path. Batch mode does not currently call `ReadPeptideIndex()` at all
(`PiStrategy::initialize` explicitly notes indexing is "loaded lazily... inside
`CometSearch::RunSearch` -> `SearchPeptideIndex`" -- but the batch overload never
calls it).

**Fix**: have batch mode call `CometPeptideIndex::ReadPeptideIndex()` once (same as
RTS), then reuse the thread-local `SearchPeptideIndex(Query*, bool*)` per query
(parallelized per finding #2). This eliminates the separate disk-streaming
implementation, the per-entry `fread` syscall overhead, and the code-duplication
maintenance burden of keeping two scoring loops in sync (they already have
drifted -- e.g. the file-based path recomputes `iWhichQuery` per peptide via
`BinarySearchMass` on `queries`, while the in-memory path binary-searches
`g_pvDBIndex` directly per query; see also the decoy handling difference in
finding #6).

## 4. Dead computation in the per-result output loop (small but free win, RTS hot path)

**Status: Done.** The `iMinLength` block was deleted outright (it was never read).
Verified with a full build and the 19-test unit suite.

`CometSearchManager::DoSingleSpectrumSearchMultiResults` (`CometSearchManager.cpp:2634-2819`)
loops over `takeSearchResultsN` results to build the return payload. Inside that
loop:

```cpp
int iMinLength = g_staticParams.options.peptideLengthRange.iEnd;
for (int x = 0; x < iSize; ++x)                    // CometSearchManager.cpp:2812
{
   int iLen = (int)strlen(pOutput[x].szPeptide);
   if (iLen == 0)
      break;
   if (iLen < iMinLength)
      iMinLength = iLen;
}
```

`iMinLength` does not depend on `iWhichResult` (it scans all `iSize` results every
time), and it is never read anywhere after being computed -- grepping the rest of
the function and file shows no further use. This is an O(`takeSearchResultsN` x
`iSize`) `strlen`-heavy computation that produces a value nobody consumes, running
once per returned PSM on every single-spectrum RTS call.

**Fix**: delete the block. It has no observable effect on output; removing it
saves `takeSearchResultsN * iSize` `strlen()` calls per spectrum (bounded by
`iNumStored`, but still pure waste at RTS latencies).

## 5. Full-size stack scratch arrays and `memset` per candidate, sized for `MAX_PEPTIDE_LEN` regardless of actual peptide length

**Status: Done**, via the first fix option below (right-sizing the `memset`
calls); the array declarations themselves were left as compile-time-sized stack
locals (the second, longer-term fix option -- moving them into the thread-local
scratch pool -- was not attempted). For both the target and decoy branches, the
single full-array `memset` was replaced with a loop over the actually-used
`(charge, ionSeries)` pairs, each clearing only the `iLenMinus1 * (VMODS + 2)`
contiguous sub-range that `XcorrScoreI()` (the sole consumer) can ever read for
that `(charge, ionSeries, len)` combination -- verified by inspecting
`XcorrScoreI`'s read loops, which use the identical `[1..usiMaxFragCharge] x
[0..iNumIonSeriesUsed) x [0..iLenPeptideMinus1)` bounds. For a typical tryptic
peptide this cuts the per-candidate memset from ~142 KB to on the order of 1-4 KB.
Validated with the full suite; T20's exact xcorr/AScore expected-value assertions
passing is strong evidence no out-of-range memory is read post-change.

`AnalyzePeptideIndex(Query*, ...)` (`CometSearch.cpp:2064-2237`) declares, on the
stack, for every candidate:

```cpp
unsigned int uiBinnedIonMasses[MAX_FRAGMENT_CHARGE + 1][NUM_ION_SERIES][MAX_PEPTIDE_LEN][VMODS + 2];
```

With `MAX_FRAGMENT_CHARGE=5`, `NUM_ION_SERIES=7`, `MAX_PEPTIDE_LEN=51`,
`VMODS=15` (`CometSearch/core/Constants.h:24,47,62`, `CometData.h:29`), this array
is `6 * 7 * 51 * 17 = 36,414` `unsigned int` entries (~142 KB), fully `memset` to
zero at `CometSearch.cpp:2240` on every call regardless of the actual peptide
length (`iLenMinus1`, typically 8-20 for tryptic peptides -- a small fraction of the
51-length dimension actually gets written in the two passes that follow). When
`iDecoySearch` is enabled, a second, equally-sized `uiBinnedIonMassesDecoy` array
and its own full `memset` are added (`CometSearch.cpp:2389,2516`), roughly doubling
the per-candidate stack traffic to ~284 KB.

This pattern also exists in the batch (non-RTS) scoring path, so it isn't new to
PI_DB, but it directly affects RTS tail latency because `AnalyzePeptideIndex` is
called once per mass-tolerance candidate, and every call pays this fixed cost
independent of peptide length.

**Fix options** (in order of effort):
- Size the working dimension by `iLenMinus1` instead of `MAX_PEPTIDE_LEN` -- e.g.
  only `memset` the `[0..iLenMinus1)` slice actually used in this call, rather than
  the full compile-time bound. This alone turns the memset cost from O(51) to
  O(actual peptide length) in the length dimension.
- Longer term, move these arrays into the thread-local scratch pool
  (`SearchMemoryPool`/`_ppbDuplFragmentArr`) already used for `pbDuplFragment`, sized
  once per thread at `AllocateMemory()` time instead of re-declared (and re-zeroed)
  on the stack per call.

## 6. On-the-fly decoy generation doubles per-candidate cost for `iDecoySearch == 1` (design/config note, not a bug)

**Status: Open.** No code change -- this remains a configuration/documentation
recommendation for anyone building `.idx` files intended for RTS use.

When `g_staticParams.options.iDecoySearch` is truthy, `AnalyzePeptideIndex(Query*, ...)`
reverses the peptide and re-runs the entire ion-ladder-generation + `XcorrScoreI`
sequence a second time for every candidate (`CometSearch.cpp:2383-2613`), on top of
whatever `bDecoyPep` routing already exists for peptides embedded from a
concatenated decoy database. This mirrors the standard (non-indexed) search's
`iDecoySearch == 1` behavior and is expected when the `.idx` was built without
pre-generated decoys.

Since PI_DB/FI_DB indexes for RTS are normally pre-built once and reused across many
spectra, there is no reason to pay the on-the-fly reversed-decoy cost per spectrum
at query time: building the index with a concatenated target-decoy FASTA
(`iDecoySearch == 2`, decoys distinguished later by protein-name prefix via
`bDecoyPep`, see `CometSearch.cpp:2145-2160`) avoids doubling ion-ladder
construction and `XcorrScoreI` calls for every single candidate scored in the RTS
hot path.

**Recommendation**: document/default RTS-oriented `.idx` builds to use
`iDecoySearch == 2` (concatenated decoy FASTA) rather than `iDecoySearch == 1`, and
flag this at index-build time if a real-time-search index is being created with
`iDecoySearch == 1` set.

## 7. Minor: fragment-list reconstruction in the output loop recomputes ion ladders already built during scoring

**Status: Open.** Not attempted -- cost is bounded by `topN` and judged negligible
relative to #1 and #5.

The output-formatting loop in `DoSingleSpectrumSearchMultiResults`
(`CometSearchManager.cpp:2836-2980`) rebuilds `dBion`/`dYion` ladders and re-runs
`BIN()` per ion/charge/series for each of the top-N results, to populate
`matchedFragments` for the API caller. This duplicates work already done inside
`AnalyzePeptideIndex`/`XcorrScoreI` during scoring, but since it only runs for
`takeSearchResultsN` (bounded by `topN`, typically small) rather than every
candidate, its cost is negligible compared to findings #1 and #5. Listed for
completeness; not worth the complexity of threading scoring-time ladders through to
the output stage.

## Suggested priority order (original plan, now completed through #5)

1. #1 (`const DBIndex&`) -- trivial, zero-risk signature change, removes an
   allocation from the single hottest per-candidate loop in the RTS path. **Done.**
2. #4 (dead `iMinLength` loop) -- trivial, zero-risk deletion. **Done.**
3. #2 + #3 (parallelize and unify batch PI_DB search) -- larger change, but batch
   PI_DB search currently gets zero benefit from `iNumThreads` > 1, which is likely
   the single biggest batch-mode throughput win available in this code path.
   **Done**, plus removed the now-dead `AnalyzePeptideIndex(int, ...)` overload
   that lost its only caller as a result.
4. #5 (right-size scratch arrays / move to thread-local pool) -- moderate effort,
   most valuable if profiling shows `memset`/stack pressure as a measurable
   fraction of per-spectrum RTS latency. **Done** (right-sizing only; the
   thread-local-pool option remains open if further profiling justifies it).
5. #6 -- configuration/documentation guidance, not a code change. **Open.**
