# Cross-platform `.raw` reading: sidecar process wire protocol (design sketch)

## Status

**Design sketch only -- not implemented.** This elaborates the "long-running sidecar process"
option recorded in the Appendix of `docs/20260618_RawFileReaderMigration.md`, which was the
leading cross-platform candidate before cross-platform `.raw` support was deprioritized in favor
of the Windows-only `/clr` rewrite (`MSToolkit/src/MSToolkit/RAWReader.cpp`). Nothing here changes
Comet's current behavior; this is reference material if cross-platform `.raw` support is revisited.

`.raw` reading on Windows is unaffected by any of this and keeps using the existing `/clr` path.
This document only concerns adding the same capability on Linux/macOS, via a separate self-contained
.NET process that Comet's native core talks to over local IPC.

## Goal

Let `comet`/`comet.exe` on Linux/macOS read `.raw` files directly, with field-for-field parity to
the Windows `/clr` implementation, by talking to a companion process
(`cometrawsidecar`, a self-contained `dotnet publish` artifact built around
`ThermoFisher.CommonCore.RawFileReader`) instead of hosting the CLR in-process.

## Why a sidecar instead of in-process `nethost`/`hostfxr` hosting

Recorded for context; see the parent conversation/doc for the full comparison. The short version:
Comet's batch search has a small number of reader/preprocess threads and many scoring threads --
spectrum reading is not the hot loop, so IPC overhead is tolerable. A sidecar keeps Comet's native
build exactly as simple as it is today (no `nethost` headers, no runtime-probing logic, no conflict
with a static Linux build), and isolates faults in third-party, closed-source RawFileReader code
(which reads arbitrary, sometimes malformed, user-supplied binary files) to a separate process
instead of risking a multi-hour unattended batch search.

## Non-goals

- Not a general-purpose RPC framework. No gRPC/protobuf: those add a heavy native dependency
  (`grpc` C++ library) to the side of the build this design is specifically trying to keep simple,
  for a protocol small enough that a few hundred lines of hand-rolled framing code covers it.
- Not used on Windows. The `/clr` path stays as-is there; this protocol only exists to cover
  platforms with no C++/CLI equivalent.
- Not designed for concurrent access from multiple reader threads in v1 (see Open Questions).

## Process model

- **Comet is the IPC server; the sidecar is the IPC client.** Comet creates and binds the listening
  socket *before* spawning the sidecar, then calls `accept()` (with a startup timeout, e.g. 10s).
  This avoids the usual sidecar-pattern race (client polling/retrying until the server's socket
  exists) entirely -- the sidecar simply connects once it starts, and Comet's blocking `accept()`
  resolves the moment that happens, with a clear timeout-driven error if it doesn't.
- **Lazy start.** Comet only spawns the sidecar the first time it needs to open a `.raw` file.
  Batches with only mzML/mzXML/mgf input never spawn it at all -- no behavior change, no new
  process, no new dependency for the common case.
- **Socket path**: a Unix domain socket at a per-invocation, collision-proof path, e.g.
  `$TMPDIR/comet-raw-<pid>-<random>.sock`, passed to the sidecar as `argv[1]`.
- **Self-cleanup on disconnect.** The sidecar's main loop treats a closed/reset connection (EOF on
  read) as "shut down now," not an error to retry. This is the actual orphan-process backstop: even
  if Comet is killed ungracefully (`SIGKILL`, can't run cleanup code), the OS closes Comet's end of
  the socket when the process dies, which unblocks the sidecar's read with EOF and it exits. No
  process-group/job-object bookkeeping is required for correctness, though using one is still a
  reasonable defense-in-depth addition.
- **Graceful shutdown.** On normal completion, Comet sends `Shutdown` and waits (bounded) for the
  child to exit before returning control to the OS.

## Transport

**Unix domain socket** (`AF_UNIX`/`SOCK_STREAM`), via plain BSD sockets on the C++ side and
`System.Net.Sockets.Socket` + `UnixDomainSocketEndPoint` on the C# side (a standard, documented
.NET API since .NET Core 3.0 -- not an internal implementation detail). This is the natural,
lowest-overhead local-IPC primitive on Linux/macOS, with a symmetric, non-platform-specific API on
both sides of the protocol. (Named pipes were considered and rejected for this specific direction:
.NET's `NamedPipeServerStream` is implemented over Unix domain sockets on Linux/macOS internally
anyway, so using `AF_UNIX` directly is more explicit and avoids depending on that implementation
detail. TCP loopback was also considered -- fully portable, slightly simpler error model -- but
adds a small amount of unnecessary network-stack overhead and a port-allocation/collision concern
that a per-invocation socket path doesn't have; not used here since this protocol never needs to
run on Windows.)

## Framing

Every message (request or response) on the stream is:

```
[ 4 bytes : payload length, uint32, little-endian ]
[ N bytes : payload ]
```

The payload's first byte is the message type tag (see below). All multi-byte integers and floats
in the payload are little-endian (the native byte order on every platform Comet targets today --
called out explicitly as a protocol invariant rather than left implicit). Strings are UTF-8,
length-prefixed with a 4-byte `int32` byte count -- not null-terminated, so arbitrary file paths
and trailer text round-trip safely.

## Message types

| Tag | Name | Direction | Purpose |
|---|---|---|---|
| 0x01 | `Hello` | Comet -> sidecar | Protocol version handshake |
| 0x02 | `HelloAck` | sidecar -> Comet | Version handshake response |
| 0x10 | `OpenFile` | Comet -> sidecar | Open a `.raw` file |
| 0x11 | `OpenFileResult` | sidecar -> Comet | Open result + file metadata |
| 0x20 | `SetMSLevelFilter` | Comet -> sidecar | Mirrors `RAWReader::setMSLevelFilter` |
| 0x21 | `SetRawFilter` | Comet -> sidecar | Mirrors `RAWReader::setRawFilter`/`setRawFilterExact` (stub in v1 -- see below) |
| 0x22 | `Ack` | sidecar -> Comet | Generic ack for filter-setting calls |
| 0x30 | `ReadScan` | Comet -> sidecar | Mirrors `RAWReader::readRawFile(NULL, s, scNum)` |
| 0x31 | `ScanResult` | sidecar -> Comet | A matching spectrum's full field set |
| 0x32 | `NoMoreScans` | sidecar -> Comet | No more scans match (mirrors `readRawFile` returning `false`) |
| 0x40 | `CloseFile` | Comet -> sidecar | Close the open file |
| 0x50 | `Shutdown` | Comet -> sidecar | Terminate the sidecar |
| 0xFF | `Error` | sidecar -> Comet | Protocol-level error (malformed request, unknown tag, etc.) |

`SetRawFilter` has zero current callers anywhere in `CometSearch`/`Comet.cpp` (confirmed during the
`/clr` rewrite) -- its message shape is defined here for parity with `RAWReader`'s public API, but
the sidecar can legitimately reply `Ack` without applying anything in a first implementation,
exactly mirroring `setRawFilterExact`'s already-dead state in the production C++ today.

## Message layouts

### `Hello` / `HelloAck`

```
Hello:     [tag=0x01][int32 protocolVersion]
HelloAck:  [tag=0x02][int32 protocolVersion][int32 sidecarBuildVersion]
```

Comet sends the protocol version it was built against immediately after connecting. If
`HelloAck.protocolVersion` doesn't match, Comet aborts with a clear "sidecar binary is out of date,
reinstall/rebuild it" error instead of proceeding with potentially-incompatible framing.

### `OpenFile` / `OpenFileResult`

```
OpenFile:        [tag=0x10][string path]

OpenFileResult:  [tag=0x11]
                 [byte success]
                 if success:
                   [int32 scanCount]          -- RunHeaderEx.LastSpectrum, i.e. RAWReader::rawTotSpec
                   [string instrumentModel]   -- InstrumentData.Model
                 else:
                   [string errorMessage]
```

`getManufacturer()` is not included -- the existing `/clr` implementation always returns the
hardcoded literal `"Thermo Scientific"` and never actually queries it from the file (confirmed
during the rewrite); the sidecar doesn't need a wire field for a value that's already a client-side
constant.

### `SetMSLevelFilter` / `Ack`

```
SetMSLevelFilter:  [tag=0x20][int32 count][count x int32 MSSpectrumType]
Ack:               [tag=0x22]
```

`MSSpectrumType` values are transmitted as the same ordinal values as the native enum
(`MSToolkitTypes.h`): `MS1=0, MS2=1, MS3=2, ZS=3, UZS=4, IonSpec=5, SRM=6, REFERENCE=7,
Unspecified=8, MSX=9`. Both sides must keep this ordinal list in sync -- it's a hand-maintained
contract, same as any binary protocol's enum encoding.

### `ReadScan` / `ScanResult` / `NoMoreScans`

```
ReadScan:  [tag=0x30][int32 scNum]
```

`scNum` carries the exact same three-way semantics as `RAWReader::readRawFile`'s parameter today:
`0` = advance from the current position, `>0` = jump to that absolute scan number, `<0` = step back
one. The sidecar runs the *entire* skip-until-match loop server-side (the same logic
`EvaluateScanTokens`/`msLevelFilter`/`rawUserFilter` checking already does in `RAWReader.cpp`) and
returns either one fully-populated matching scan or `NoMoreScans` -- never a partial/no-match
response that would require Comet to loop and re-request. This is the key design choice that keeps
the protocol at one round-trip per *returned* spectrum, not one round-trip per *examined* scan
number, since a single multiplexed/zoom/SRM-filtered file could otherwise skip many scans before
finding a match.

```
NoMoreScans:  [tag=0x32]

ScanResult:   [tag=0x31]
              [int32 scanNumber]
              [float64 retentionTime]
              [byte centroidStatus]            -- 0 or 1
              [byte fileType]                  -- MSSpectrumType ordinal (see above)
              [byte activationMethod]          -- MSActivation ordinal (see below)
              [float32 bpi]
              [float64 bpm]
              [float64 tic]
              [float64 compensationVoltage]
              [float32 ionInjectionTime]
              [byte hasScanDescription]
                if set: [string scanDescription]
              [int32 spsCount][spsCount x float64 spsMass]
              [byte hasPrecursor]
                if set:
                  [int32 charge]
                  [float64 monoMz]
                  [float64 mz]
                  [float64 isoMz]
                  [int32 precursorScanNumber]
              [int32 mzEntryCount][mzEntryCount x (float64 mz, float64 monoMz)]
              [int32 zStateCount][zStateCount x (int32 charge, float64 mass)]
              [int32 peakCount][peakCount x (float64 mz, float32 intensity)]
```

`MSActivation` ordinals (`MSToolkitTypes.h`): `mstCID=0, mstECD=1, mstETD=2, mstETDSA=3, mstPQD=4,
mstHCD=5, mstIRMPD=6, mstSID=7, mstNA=8`.

`setScanNumber` (called twice in `RAWReader.cpp`, both times with the same value), `setNativeID`
(always exactly `"scan=" + scanNumber`), and `setMsLevel` (a deterministic function of `fileType`:
`MS1->1, MS2->2, MS3->3, MSX->2, default->0`) are **not** transmitted -- they're all
either redundant or mechanically derivable from `scanNumber`/`fileType` on the client side, so the
C++ client reconstructs them locally instead of paying wire bytes for them. `mzEntryCount` and
`zStateCount` are almost always `0` or `1` in practice (multiple entries only occur for MSX/unknown-
charge fallback paths) but are encoded as counted lists rather than a fixed `0`/`1` slot, matching
`Spectrum::addMZ`/`addZState` being genuinely repeatable calls in the existing C++ implementation.

### `CloseFile`, `Shutdown`

```
CloseFile:  [tag=0x40]   -- no response; next OpenFile implicitly closes any open file
Shutdown:   [tag=0x50]   -- no response; sidecar exits after receiving this
```

### `Error`

```
Error:  [tag=0xFF][string message]
```

Returned for protocol-level problems (unrecognized tag, malformed payload length, etc.) --
distinct from `NoMoreScans`, which is an expected, normal outcome of `ReadScan`, not an error.

## Mapping to the existing `RAWReader.cpp` implementation

Every field above corresponds to one of the `Spectrum::set*`/`add*` calls already validated in the
Windows `/clr` rewrite (`MSToolkit/src/MSToolkit/RAWReader.cpp:407-577`):

| Wire field(s) | Existing call |
|---|---|
| `spsCount`/`spsMass` | `s.addSPS(double)` |
| `scanDescription` | `s.setScanDescription(string)` |
| `ionInjectionTime` | `s.setIonInjectionTime(float)` |
| `hasPrecursor` block | `s.addPrecursor(MSPrecursorInfo)` (`pi.activation` is never set by `RAWReader.cpp` today -- stays `mstNA` on the client side, no wire field needed for it) |
| `mzEntryCount`/entries | `s.setMZ`/`s.addMZ(double, double mono=0)` |
| `zStateCount`/entries | `s.addZState(int, double)` |
| (client-derived) | `s.setCharge(int)` -- derivable from the precursor charge or the unknown-charge fallback, same logic already in `RAWReader.cpp` |
| `centroidStatus` | `s.setCentroidStatus(int)` |
| `activationMethod` | `s.setActivationMethod(MSActivation)` |
| `scanNumber` (client derives the second `setScanNumber` call and `setNativeID`) | `s.setScanNumber(int, bool)` x2, `s.setNativeID(const char*)` |
| `retentionTime` | `s.setRTime(float)` |
| `fileType` (client derives `setMsLevel`) | `s.setFileType(MSSpectrumType)`, `s.setMsLevel(int)` |
| `bpi`, `bpm`, `compensationVoltage`, `tic` | `s.setBPI`/`setBPM`/`setCompensationVoltage`/`setTIC` |
| `peakCount`/peaks | `s.add(double, float)` loop |

This table exists specifically so a future implementation can be checked field-by-field against
the already-validated Windows behavior, rather than re-deriving correctness from scratch.

## Open questions / not yet designed

- **Concurrent readers.** If Comet's batch search ever reads `.raw` spectra from more than one
  thread concurrently, this v1 protocol (one socket, one client, synchronous request/response)
  would serialize them. Options if this turns out to matter: multiple sidecar connections (one per
  reader thread, still one sidecar process), or request pipelining with response IDs on a single
  connection. Not designed here since it's not yet established that Comet's batch path needs
  concurrent `.raw` access at all.
- **Performance validation.** The IPC/serialization overhead this design accepts as "tolerable
  because scoring dominates" has not been benchmarked against this codebase's actual fragment-index
  search throughput. Worth measuring before committing to this design over in-process hosting.
- **Sidecar binary distribution.** Not addressed here: how `cometrawsidecar` itself ships (separate
  download vs. bundled with the Linux/macOS release tarball), and how its self-contained .NET
  publish target/runtime identifier (linux-x64, osx-x64, osx-arm64, ...) is selected at build time.
